SECTION 2 — Why the DROP INDEX line is stripped
===============================================================================

The line:

    DROP INDEX "Product_embedding_cosine_idx";

is a FALSE POSITIVE from `prisma migrate diff`. The Prisma schema DSL cannot
express pgvector IVFFlat indexes — they live in raw SQL inside migration
20260426150000_add_pgvector_and_product_embedding/migration.sql. Because
Prisma never sees the index in schema.prisma but sees it in the live DB,
every `migrate diff` against the live datasource emits a DROP for it.

What would happen if we accepted the DROP:

  - The IVFFlat index `Product_embedding_cosine_idx` would be physically
    removed from Postgres on the next `prisma migrate deploy`.
  - The recommendation engine's pgvector cosine-distance retrieval (Phase 3
    Stage 2 of the six-stage pipeline) would degrade from O(log N) IVFFlat
    lookup to a sequential scan over every Product row in the shop.
  - On the dev shop's 2,632 products, every recommendation call would
    sequentially scan ~2.6K vectors. Tolerable in dev. Catastrophic on a
    50K-product merchant.
  - The damage is silent — no error, just slow recs. We would only catch it
    via Phase 3 latency metrics, possibly weeks after deploy.

Therefore: the DROP line is omitted from the committed migration. The
omission is documented in the migration SQL header (lines 1-21 of
prisma/migrations/20260503130000_add_tagging_review_and_jobs/migration.sql)
so a future reader of the migration alone (without this artifact) sees the
discipline applied.

The same strip discipline was applied in PR-D D.1
(20260501023350_add_customer_profile_schema/migration.sql) — that file is
the precedent and its header comment is the template.

SECTION 3 — Two additions beyond the diff output
===============================================================================

The committed migration SQL contains two things that `prisma migrate diff`
does NOT emit:

(a) Two partial unique indexes on TaggingJob. Prisma DSL cannot express a
    WHERE clause on @@unique, so partial uniques are added in raw SQL after
    the auto-generated indexes. The same pattern is used by
    20260427152535_add_catalog_sync_job/migration.sql for CatalogSyncJob's
    "at most one RUNNING per shop" partial unique index.

    Index 1:  TaggingJob_shopDomain_productId_queued_uniq
              UNIQUE on (shopDomain, productId)
              WHERE status = 'QUEUED' AND productId IS NOT NULL
              Purpose: enforces QUEUED-only dedup for SINGLE_PRODUCT and
                       MANUAL_RETAG kinds at the DB layer.

    Index 2:  TaggingJob_shopDomain_initial_backfill_active_uniq
              UNIQUE on (shopDomain)
              WHERE kind = 'INITIAL_BACKFILL' AND status IN ('QUEUED', 'RUNNING')
              Purpose: at most one INITIAL_BACKFILL per shop in flight.

(b) One-time data backfill UPDATE for ProductTag.status. Predicate:

        UPDATE "ProductTag" SET "status" = 'APPROVED' WHERE "source" = 'HUMAN';

    Pre-existing rows with source='HUMAN' represent merchant-marked-reviewed
    tags from feature 005d (the existing api.products.$id.mark-reviewed
    route). These are implicitly approved; flipping them to status='APPROVED'
    in the migration preserves that history across the schema change.

    Predicate is EXACTLY source='HUMAN'. Risk #3 from the planning round
    flagged that a broader UPDATE would silently mark un-reviewed AI tags
    as APPROVED, which would let unreviewed AI suggestions leak into the
    Stage 4 merchant-signal injection ranking boost. The predicate is
    deliberately narrow.

    AI rows stay PENDING_REVIEW. Pre-existing RULE rows also stay
    PENDING_REVIEW; they get lifted to APPROVED on the next rule-engine
    write per product (Item 8 of PR-2.1 changes rule-engine.server.ts to
    write status='APPROVED' on RULE-source tags going forward).
